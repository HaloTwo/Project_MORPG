Project MORPG 현재 구현 정리
작성일: 2026-05-03

1. 프로젝트 방향

이 프로젝트는 Unity 클라이언트가 DB를 직접 접근하지 않고,
C++ TCP 서버를 통해 계정과 캐릭터 데이터를 검증/저장하는 구조로 만든 MORPG 프로젝트입니다.

현재 목표는 거대한 온라인 RPG를 한 번에 만드는 것이 아니라,
온라인 RPG의 기본 진입 흐름인 로그인, 회원가입, 캐릭터 슬롯, 캐릭터 생성/삭제, 게임 입장 흐름을 먼저 완성하는 것입니다.

전체 흐름은 아래와 같습니다.

Unity Client
-> C++ TCP Server
-> MariaDB

Unity는 화면, 입력, 캐릭터 선택 UI를 담당합니다.
C++ 서버는 로그인, 회원가입, 캐릭터 생성, 캐릭터 삭제, 게임 입장 가능 여부를 판단합니다.
MariaDB는 계정과 캐릭터 데이터를 실제로 저장합니다.


2. 현재 서버 구현 방식

서버는 C++17과 Winsock 기반 TCP 서버입니다.
현재는 Blocking TCP 구조와 Text Protocol을 사용합니다.

처음부터 IOCP나 바이너리 패킷으로 만들면 구조가 너무 커지기 때문에,
우선은 눈으로 확인하기 쉬운 텍스트 명령 방식으로 Unity와 서버의 요청/응답 흐름을 검증했습니다.

현재 서버가 받는 주요 명령은 아래와 같습니다.

LOGIN 아이디 비밀번호
REGISTER 아이디 비밀번호
CREATE_CHARACTER 계정ID 슬롯번호 직업
DELETE_CHARACTER 계정ID 캐릭터ID
ENTER_GAME 캐릭터ID
PING
QUIT

예를 들어 Unity에서 test_user / 1234로 로그인하면,
Unity는 LOGIN test_user 1234 같은 텍스트 명령을 서버로 보냅니다.
서버는 MariaDB의 accounts 테이블에서 계정을 찾고,
맞으면 LOGIN_OK와 캐릭터 목록을 Unity로 돌려줍니다.


3. 서버 코드 역할

Server_CPP/src/main.cpp
- 서버 시작 지점입니다.
- 포트, DB 접속 정보, DB 비밀번호 입력 처리를 준비합니다.
- MariaDbAccountRepository, AuthService, TcpServer를 생성해 서버를 실행합니다.
- DB 비밀번호는 코드에 박아두지 않고 실행 시 입력하거나 인자로 받을 수 있게 했습니다.

Server_CPP/src/net/TcpServer.*
- TCP 포트를 열고 클라이언트 접속을 기다립니다.
- 클라이언트가 접속하면 ClientSession으로 넘겨 실제 요청 처리를 맡깁니다.

Server_CPP/src/net/ClientSession.*
- 클라이언트 한 명의 요청을 처리하는 코드입니다.
- LOGIN, REGISTER, CREATE_CHARACTER, DELETE_CHARACTER, ENTER_GAME 같은 명령을 읽습니다.
- 명령을 해석한 뒤 AuthService에 실제 처리를 요청합니다.

Server_CPP/src/protocol/PacketCodec.*
- 서버가 받은 텍스트 명령을 ClientCommand로 분리합니다.
- 서버 응답 문자열을 만듭니다.
- 예: LOGIN_OK, LOGIN_FAIL, CHARACTER_LIST, CREATE_CHARACTER_OK, DELETE_CHARACTER_OK

Server_CPP/src/service/AuthService.*
- 로그인, 회원가입, 캐릭터 생성, 캐릭터 삭제, 게임 입장 흐름을 제어하는 서비스 계층입니다.
- DB를 직접 알지 않고 IAccountRepository 인터페이스를 통해 저장소 기능을 사용합니다.
- 이렇게 분리하면 나중에 MariaDB가 아니어도 저장소 구현만 바꾸면 됩니다.

Server_CPP/src/repository/IAccountRepository.h
- 계정 저장소가 반드시 구현해야 하는 기능을 정의한 인터페이스입니다.
- 로그인 조회, 회원가입, 캐릭터 조회, 캐릭터 생성, 캐릭터 삭제 기능을 정의합니다.

Server_CPP/src/repository/MariaDbAccountRepository.*
- 실제 MariaDB에 접속해서 accounts, characters, character_skills 테이블을 조회/수정합니다.
- 로그인 계정 확인, 중복 회원가입 방지, 캐릭터 생성, 캐릭터 삭제를 처리합니다.
- 캐릭터 삭제는 account_id와 character_id를 함께 조건으로 걸어서,
  다른 계정 캐릭터를 지우지 못하게 했습니다.

Server_CPP/db/schema.sql
- MariaDB에 만들 테이블 구조가 들어 있습니다.
- accounts, characters, character_skills, inventory_items, equipment 테이블을 생성합니다.
- test_user / 1234 테스트 계정도 기본으로 넣습니다.


4. Unity 코드 역할

Assets/3.Script/Server/Network/NetworkManager.cs
- Unity 쪽 네트워크 진입점입니다.
- 서버 연결, 패킷 전송, 수신 큐 처리, 서버 끊김 팝업을 담당합니다.
- 서버가 꺼져 있으면 “서버가 끊겼습니다.” 팝업을 띄우고 확인 시 게임을 종료합니다.

Assets/3.Script/Server/Network/TcpServerConnection.cs
- 실제 TCP 연결과 송수신을 담당합니다.
- 서버에서 온 텍스트 응답을 ServerTextProtocol로 넘겨 Unity 패킷으로 변환합니다.

Assets/3.Script/Server/Network/ServerTextProtocol.cs
- Unity Packet을 서버 텍스트 명령으로 바꿉니다.
- 서버 응답 텍스트를 Unity Packet으로 다시 바꿉니다.
- 예: DeleteCharacterRequestPacket -> DELETE_CHARACTER 계정ID 캐릭터ID

Assets/3.Script/Server/Network/PacketDispatcher.cs
- 수신된 패킷을 종류별 이벤트로 나눠 UI나 게임 코드에 전달합니다.
- 로그인 응답, 회원가입 응답, 캐릭터 목록, 캐릭터 생성 응답, 캐릭터 삭제 응답 등을 분배합니다.

Assets/3.Script/Client/Login/LoginSceneController.cs
- 로그인 화면을 코드로 생성합니다.
- 로그인은 메인 패널에서 처리합니다.
- 회원가입은 별도 팝업 창을 띄워 아이디/비밀번호를 입력하게 했습니다.
- 로그인 실패 시 “아이디 또는 비밀번호가 틀렸습니다.”처럼 사용자가 이해하기 쉬운 메시지를 보여줍니다.

Assets/3.Script/Client/Character/CharacterSelectSceneController.cs
- 캐릭터 선택 화면을 생성합니다.
- 계정당 3개의 슬롯을 보여줍니다.
- 빈 슬롯은 전사/궁수/도적 생성 버튼을 보여줍니다.
- 생성된 캐릭터는 입장 버튼과 삭제 버튼을 보여줍니다.
- 삭제 요청은 서버로 보내고, 서버가 성공 응답을 줘야 UI에서 제거합니다.

Assets/3.Script/Client/Character/CharacterSession.cs
- 현재 로그인한 accountId, 캐릭터 목록, 선택 캐릭터를 저장합니다.
- 씬이 바뀌어도 선택 캐릭터 정보를 유지하기 위한 클라이언트 세션 저장소입니다.
- 캐릭터 삭제 성공 시 목록에서 제거합니다.

Assets/3.Script/Client/Core/GameManager.cs
- 게임 씬에서 선택한 캐릭터 데이터를 로컬 플레이어에 적용합니다.
- 현재는 직업에 따라 색상을 다르게 보여주는 방식으로 구분합니다.
- 게임 씬에서 캐릭터 선택창으로 돌아가는 버튼도 생성합니다.

Assets/3.Script/Client/UI/RuntimeUiFactory.cs
- Canvas, Panel, Text, Button, InputField를 코드에서 생성하는 유틸입니다.
- 로그인, 회원가입, 캐릭터 선택 UI가 이 코드를 사용합니다.


5. 현재 동작 순서

1. MariaDB가 실행되어 있어야 합니다.
2. C++ 서버를 실행합니다.
3. 서버는 MariaDB 접속 비밀번호를 입력받거나 실행 인자로 받습니다.
4. Unity에서 LoginScene을 실행합니다.
5. Unity가 서버에 TCP 연결을 시도합니다.
6. 로그인 또는 회원가입을 합니다.
7. 서버가 MariaDB에서 계정을 확인하거나 새 계정을 추가합니다.
8. 서버가 캐릭터 목록을 Unity로 보냅니다.
9. Unity는 CharacterSelectScene으로 이동합니다.
10. 캐릭터 슬롯 3칸을 보여줍니다.
11. 빈 슬롯에서 직업을 선택하면 서버에 CREATE_CHARACTER 요청을 보냅니다.
12. 서버가 DB에 캐릭터를 저장하고 CREATE_CHARACTER_OK를 보냅니다.
13. Unity는 새 캐릭터를 슬롯에 표시합니다.
14. 삭제 버튼을 누르면 DELETE_CHARACTER 요청을 서버로 보냅니다.
15. 서버가 DB에서 해당 계정의 캐릭터를 삭제하고 DELETE_CHARACTER_OK를 보냅니다.
16. Unity는 해당 슬롯을 빈 슬롯으로 다시 표시합니다.
17. 입장 버튼을 누르면 GameScene으로 이동합니다.
18. GameScene에서 캐릭터 선택 버튼을 누르면 CharacterSelectScene으로 돌아갑니다.


6. 현재까지 해결한 문제와 선택한 이유

서버 없이 Unity만으로 로그인/회원가입을 만들면 구현은 빠르지만,
클라이언트가 데이터를 마음대로 조작할 수 있습니다.
그래서 계정과 캐릭터 판단은 서버에서 처리하도록 했습니다.

Unity에서 MariaDB에 직접 접속하는 방식도 가능은 하지만,
DB 계정 정보가 클라이언트에 노출되고 보안상 좋지 않습니다.
그래서 Unity -> Server -> DB 구조로 분리했습니다.

처음부터 IOCP 서버를 만들면 네트워크 구조가 너무 커져서,
현재 단계에서는 Blocking TCP와 Text Protocol로 빠르게 흐름을 검증했습니다.
나중에 구조가 안정되면 IOCP와 Binary Packet으로 바꾸기 좋게 계층을 나눴습니다.

캐릭터 슬롯은 리스트 순서로 보여주면 삭제 후 슬롯이 밀릴 수 있습니다.
그래서 DB에 slot_index를 저장하고, Unity도 slot_index 기준으로 3칸을 표시하게 했습니다.

DB 비밀번호는 코드에 하드코딩하지 않았습니다.
서버 실행 시 콘솔 입력이나 실행 인자로 전달하게 해서,
GitHub에 비밀번호가 올라가지 않도록 했습니다.
